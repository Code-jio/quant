Three approaches to get the live trading loop working

Goal: strategy → signal → risk → order → fill → position, end-to-end on rb2510

A

CLI-first，前后端分离验证（推荐）

先不管前端 UI，专注打通 CLI 路径：python main.py -m live -c config/config_production.json

终端日志输出全链路：tick → bar → signal → order → fill。等引擎跑通后，再用前端 Dashboard 做可视化监控。

Pros

  • 最快到达"能下单"状态
  • 调试链路短，出问题容易定位
  • 前端改动独立排期，不阻塞引擎验证

Cons

  • 首轮验证只有终端日志，体验简陋
  • 前端监控要等第二轮才能用

工作量：约 3-4 天（配置 + 引擎修复 + 端到端验证）

B

前后端一起上，完整 Web 控制台

同时修复 CLI 引擎路径 + 前端 TradingPanel/Dashboard，用户通过浏览器完成从登录到下单的全部操作。

Pros

  • 一次性交付完整可用产品
  • 前端实时订单/持仓/日志面板提升信心

Cons

  • 工作量大，约 7-10 天
  • 前端和后端问题交织，调试复杂
  • WebSocket 断线重连等前端问题可能拖慢进度

工作量：约 7-10 天

C

最小 CLI + 精简监控页面

打通 CLI 引擎，同时做一个极简的单页监控（只读：当前持仓 + 最近订单 + 日志流），不做完整的交易面板。

Pros

  • 有基本的可视化监控，不依赖纯终端
  • 前端范围小，不容易出 bug

Cons

  • 仍然需要做前后端联调
  • 后续需要补全交易面板时会有部分返工

工作量：约 5-6 天
